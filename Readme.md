## ✨ Features

- **Structured Roadmap** — know exactly what to study and in what order
- **Progress tracking** per topic — `Not Started` → `Active` → `Completed`
- **Live progress bar** and header stats that update as you mark topics done
- **4–5 resources per pattern** — best video + visual guide + code example + blog
- **Add your own links** to any topic — stored in your browser
- **25 interview questions** with frequency ratings, company tags, expandable hints
- **Zero setup** — one `.html` file, works offline, no login needed
- **LocalStorage persistence** — your progress saved across sessions

## 🧩 Design Patterns — Complete Reference

Every pattern includes: best YouTube lecture + Refactoring.Guru visual guide + Baeldung code walkthrough + GitHub code example.

### 🧱 Fundamentals (Study First)
| # | Topic | Why it matters |
|---|---|---|
| 1 | OOP — 4 Pillars | Foundation of every LLD interview |
| 2 | SOLID Principles | Must-know for clean, extensible code |
| 3 | Composition over Inheritance | Avoids brittle class hierarchies |
| 4 | UML Class Diagrams | Draw before you code |
| 5 | Interfaces vs Abstract Classes | Know when to use each |
| 6 | Concurrency & Thread Safety | Race conditions, deadlocks, locks |

### 🏗️ Creational Patterns
| # | Pattern | One-line idea |
|---|---|---|
| 1 | Singleton | One global instance — Logger, Config, DB Pool |
| 2 | Factory Method | Let subclasses decide what to create |
| 3 | Abstract Factory | Factory of factories — UI toolkits |
| 4 | Builder | Step-by-step object construction — fluent API |
| 5 | Prototype | Clone instead of reconstruct |
| 6 | Object Pool | Reuse expensive objects — DB connections |

### 🔗 Structural Patterns
| # | Pattern | One-line idea |
|---|---|---|
| 1 | Adapter | Wrap incompatible interfaces |
| 2 | Decorator | Add behaviour dynamically — Java I/O |
| 3 | Facade | Simple interface to a complex subsystem |
| 4 | Proxy | Surrogate — lazy init, access control, caching |
| 5 | Composite | Tree structures — file system, UI trees |
| 6 | Bridge | Decouple abstraction from implementation |
| 7 | Flyweight | Share fine-grained objects to save memory |

### 🎭 Behavioural Patterns
| # | Pattern | One-line idea |
|---|---|---|
| 1 | Observer | Notify subscribers on state change |
| 2 | Strategy | Swap algorithms at runtime |
| 3 | State | Behaviour changes with internal state — FSM |
| 4 | Command | Encapsulate request — enables undo/redo |
| 5 | Template Method | Algorithm skeleton in base class |
| 6 | Iterator | Sequential access without exposing structure |
| 7 | Chain of Responsibility | Pass request along handler chain — middleware |
| 8 | Mediator | Centralise object communication |
| 9 | Memento | Capture and restore object state |
| 10 | Visitor | Add operations without modifying objects |
| 11 | Interpreter | Grammar + interpreter — SQL, regex |
